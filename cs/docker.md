Docker 이미지와 컨테이너에 대해
 

Docker를 처음 공부하다 보면 이미지와 컨테이너라는 말이 자주 나옵니다.

 

이미지는 컨테이너를 만들기 위한 읽기 전용의 실행 템플릿이고, 컨테이너는 이미지를 기반으로 실제 실행된 인스턴스 자체를 의미합니다. 

 

조금 더 쉽게 말하면 이미지는 실행 전의 준비물이고, 컨테이너는 실행 중인 결과물이라고 볼 수 있어요.

Dockerfile을 작성학고, 이를 빌드하면 이미지가 되는 것이고, 이 이미지를 기반으로 컨테이너를 실행한다고 보면 됩니다.

즉, 이미지는 실행 전 상태, 컨테이너는 실행된 상태라고도 볼 수 있죠.

 

이미지
 

조금 더 상세히 설명해 볼게요.

Docker 이미지는 컨테이너를 실행하기 위해 필요한 파일과 설정을 묶어둔 것입니다. 이미지 안에는 여러 가지 것들이 포함되지만 대표적인 것들은 아래와 같아요.

보시면 알겠지만 이미지의 핵심은 단순히 소스 코드만 담고 있는 게 아닌 애플리케이션 단위로 실행하기 위한 파일 시스템과 실행 설정을 포함하고 있어요.

 

이미지는 실행 중인 상태가 아닙니다. 앞서 말했듯이 이미지는 실제 실행하기 위한 기본 설정에 가깝죠.

 

이러한 이미지는 기본적으로 읽기 전용입니다.

이미지는 원본 템플릿이고, 이를 읽어서 컨테이너를 만드는 기준점으로 삼는 것이죠.

때문에 컨테이너 내부에서 파일을 수정하더라도 원본 이미지가 직접 바뀌진 않습니다.

이미지는 읽기 전용이기에 수정되지 않으니까요.

 

때문에 추후 같은 이미지로 새로운 컨테이너를 만들면 기존 이미지를 따라서 컨테이너가 구성됩니다.

즉 이미지는 원본이고, 컨테이너는 이 원본을 기반으로 실행된 각각의 개별 환경인 것이죠.

때문에 한 이미지에 대해 여러 컨테이너를 여는 것도 당연히 가능합니다.

 

 

컨테이너
 

이번엔 컨테이너에 대해 더 살펴보도록 하죠.

 

컨테이너는 앞서 말했듯이 이미지를 기반으로 실제 실행된 인스턴스입니다.

이번엔 예시와 함께 설명해 볼게요.

 

docker run -d -p 8080:80 --name my-nginx nginx
 

이 명령어는 nginx 이미지를 기반으로 my-nginx라는 컨테이너를 만들고, 백그라운드에서 실행하며, 호스트의 8080 포트를 컨테이너의 80 포트와 연결한다는 의미입니다.

 

여기서 주목해야 할 점은 nginx는 이미지의 이름, my-nginx는 컨테이너의 이름이에요. 앞서 말했듯 이미지를 기반으로 새로운 컨테이너를 하나 만들어서 실행시켰다고 보면 됩니다.

 

컨테이너를 처음 보면 작은 가상머신처럼 느껴질 수 있습니다. 

하지만 정확히는 컨테이너는 격리된 환경에서 실행되는 프로세스에 더 가깝습니다.

 

Docker 공식 문서에서도 컨테이너를 호스트에서 실행되는 일종의 프로세스라고 설명하고, Docker run으로 실행되는 컨테이너 프로세스는 자체 파일 시스템, 자체 네트워킹, 호스트와 분리된 프로세스 트리를 가진다고 설명하고 있어요.

즉 정리하자면 컨테이너는, 전용 파일 시스템, 프로세스 공간, 네트워크 환경, 환경 변수, 호스트 이름 등의 독립되어 있는 실행 환경(값)들을 가지고 있습니다.

 

하지만 이러한 각각의 컨테이너가 하나의 완전한 컴퓨터 하나를 새로 부팅한다고 보진 않아요. 기본적으로 컨테이너는 호스트의 OS 커널을 공유하기 때문이죠. Docker 공식 문서상에서도 컨테이너는 서로 격리되지만 OS 커널은 공유한다고 되어 있습니다. 즉 VM(Virtual Machine)은 자체 커널을 포함한 전체 운영체제를 실행하지만, 컨테이너는 실행에 필요한 파일을 가진 격리된 일종의 프로세스에 가까운 거죠.

 

그럼 말이 나온 김에, VM과 컨테이너의 차이점에 대해 더 알아보죠.

 

VM은 컴퓨터 안에 또 하나의 컴퓨터를 띄우는 방식에 가깝습니다. 

VM 안에는 게스트 OS, 커널, 드라이버, 라이브러리, 애플리케이션 등이 들어가고 이는 하나의 컴퓨터, 또는 하나의 시스템이라 볼 수 있기 때문이죠. 반면 컨테이너는 핵심인 OS 커널을 공유하고 있어요. 때문에 일종의 프로세스에 가깝다고 하는 겁니다. 세세한 차이점에 대해선 표로 정리해 둘게요.

 

실행 단위	운영체제 전체	격리된 프로세스
커널	VM마다 별도 커널	호스트 커널 공유
무게	상대적으로 무거움	상대적으로 가벼움
시작 속도	상대적으로 느림	상대적으로 빠름
격리 방식	하드웨어 가상화 기반	OS 커널 기능 기반
용도	완전한 OS 환경 분리	애플리케이션 실행 환경 분리
 

 

이미지와 컨테이너
 

그럼 다시 돌아와서 이미지와 컨테이너의 관계에 대해 다시 살펴볼게요.

 

'이미지는 원본, 컨테이너는 이를 기반으로 실행한 일종의 프로세스, 그리고 이러한 하나의 원본으로 여러 개의 프로세스(컨테이너)를 실행할 수 있다.'라고 볼 수 있습니다.

 

예시와 함께 다시 얘기해 볼게요.

 

가령 nginx 이미지가 하나 있다고 볼게요. 이 이미지 하나로 여러 개의 컨테이너를 다음과 같이 실행할 수 있습니다.

 

docker run -d --name nginx-1 nginx
docker run -d --name nginx-2 nginx
docker run -d --name nginx-3 nginx
 

그리고 구조는 다음과 같은 느낌이죠.

docker run -d --name nginx-1 nginx
docker run -d --name nginx-2 nginx
docker run -d --name nginx-3 nginx
 

또한 각 컨테이너는 독립적이기에, nginx-1가 중지되든, 삭제되든 다른 nginx-2나 nginx-3등에 영향이 가지 않습니다. 

또한 이러한 컨테이너들을 수정하더라도 원본인 nginx라는 이미지는 변함이 없죠.

 

레이어
 

이제 이미지와 컨테이너에 대해 개념을 익혔으니, 좀 더 자세한 내용을 다뤄보려고 합니다. 그중에서도 우선 레이어에 대해 설명해 볼게요.

 

Docker 이미지는 하나의 파일처럼 보이지만 내부적으론 여러 개의 레이어로 구성되어 있어요. 가령 다음과 같은 Dockerfile이 있다고 가정해 볼게요.

 

FROM eclipse-temurin:17-jre

WORKDIR /app

COPY build/libs/app.jar app.jar

ENTRYPOINT ["java", "-jar", "app.jar"]
 

이 Dockerfile로 이미지를 만들면 대략 다음과 같은 레이어 구조가 생깁니다.

eclipse-temurin:17-jre 기반 레이어
   ↓
/app 작업 디렉터리 설정 레이어
   ↓
app.jar 복사 레이어
   ↓
컨테이너 실행 명령어 설정 레이어
 

이렇게 이미지를 여러 개의 레이어로 나누는 이유는 효율성 때문이에요.

가령 공통으로 사용되는 레이어는 재사용할 수 있습니다.

 

각 프로젝트별로 쓰이는 이미지가 있을 때, 이 이미지 모두가 Java 17 기반으로 동작한다고 하면, 모두 같은 eclipse-temurin:17-jre 기반 레이어를 가지고 있을 겁니다. 그럼 이 레이어에 대해서 모든 이미지가 재사용하여 쓸 수 있어요.

 

또한 이미지를 다시 빌드할 때 변경되지 않은 레이어는 캐시를 사용할 수 있습니다.

 

즉 각 층별로 나눠놓고, 변경된 부분만 추가로 실행하거나, 공통으로 쓰이는 부분은 재사용하는 등 최적화를 위해 쪼개서 이용한다고 보면 될 것 같아요.

 

또한 기본적으로 Dockerfile은 위에서 아래로 실행됩니다. 그리고 이 각 단계는 캐시 대상이 될 수 있죠. 그렇기 때문에 자주 바뀌지 않는 것은 최대한 위로 두고, 자주 바뀌는 것은 최대한 아래쪽에 두어서 용이하게 쓰게끔 할 수 있습니다.

 

컨테이너 수정
 

그럼 레이어의 개념에 대해 이해했으니 다음은 어떻게 컨테이너에서 파일 수정이 가능한가에 대해 살펴볼게요.

 

위에선 컨테이너는 이미지를 기반으로 새롭게 생성한 각각의 프로세스다.라고 설명했습니다. 좀 더 자세히 설명하자면,

컨테이너는 기본적으로 이미지 위에 쓰기 가능한 레이어를 하나 추가해서 실행됩니다.

그리고 이러한 쓰기 가능한 레이어는 각 컨테이너마다 가지고 있고, 고유하기 때문에 서로 영향을 미치지 않죠.

 

같은 이미지를 기반으로 하더라도, 각각의 컨테이너마다 고유의 쓰기가 가능한 레이어를 가지게 되고, 각각 공유되지 않기 때문에 독립된 상태를 가질 수 있는 것입니다.

 

그럼 컨테이너를 삭제하게 되면 어떻게 될까요?

당연하게도 컨테이너를 삭제하면 쓰기 레이어도 함께 사라질 수 있습니다. 때문에 컨테이너로 띄운 DB에 데이터를 저장해 뒀다가, 이 컨테이너를 삭제하게 되면 쓰기 레이어에 포함되어 있던 데이터 정보들이, 컨테이너 삭제와 함께 같이 날아가게 되죠.  때문에 컨테이너에선 종종 Volume을 쓰기도 합니다. 

 

Volume
 

컨테이너는 언제든지 만들고 삭제할 수 있는 실행 단위입니다.

그래서 컨테이너 내부에 중요한 데이터를 저장하는 건 좋지 못합니다. 이때 데이터를 컨테이너와 분리하기 위해 사용되는 것이 Volume입니다.

 

Volume은 Docker가 관리하는 별도의 데이터 저장 공간입니다.

컨테이너 내부 특정 경로와 volume을 연결하면, 그 경로에 저장되는 데이터는 컨테이너 내부 쓰기 레이어가 아니라 volume에 저장됩니다.

 

예를 들어 MySQL 컨테이너를 실행할 때 다음처럼 실행합니다.

docker run -d \
  --name mysql-container \
  -e MYSQL_ROOT_PASSWORD=1234 \
  -v mysql-data:/var/lib/mysql \
  mysql:8.0
 
 

이렇게 하면 Docker Volume인 mysql-data를 컨테이너 내부의 /var/lib/mysql 경로에 연결할 수 있어요. 

이렇게 하면 MySQL 데이터가 컨테이너 내부의 쓰기 레이어가 아닌 volume에 저장되게 됩니다.

 

이때 컨테이너를 삭제해도 volume은 기본적으로 살아있습니다. 즉 컨테이너의 삭제가 데이터의 삭제로 이어지는 것을 막을 수 있는 것이죠.

 

Dockerfile
 

Dokcerfile은 이미지를 만들기 위한 파일입니다.

 

Dockerfile에는 이미지가 어떤 기반에서 시작하고, 어떤 파일을 복사하고, 어떤 명령어를 실행할지 적습니다.  Dockerfile은 사용자가 이미지를 조립하기 위해 명령줄에서 호출할 수 있는 모든 명령을 담은 텍스트 문서라고 볼 수 있는 거죠.

 

좀 더 자세히 알아보기 위해 예시와 함께 설명해 볼게요.

 

예를 들어 Spring Boot 애플리케이션을 Docker 이미지로 만들기 위한 Dockerfile은 다음과 같습니다.

FROM eclipse-temurin:17-jre

WORKDIR /app

COPY build/libs/app.jar app.jar

EXPOSE 8080

ENTRYPOINT ["java", "-jar", "app.jar"]
 
FROM
 
FROM eclipse-temurin:17-jre
 
FROM은 기반 이미지를 지정합니다.

이 Dockerfile은 eclipse-temurin:17-jre 이미지를 기반으로 새 이미지를 만들겠다는 뜻입니다.

Spring Boot 애플리케이션은 Java가 필요하니, Java17 버전을 FROM으로 지정한 셈이죠.

 
WORKDIR
 

WORKDIR /app
 
WORKDIR은 컨테이너 내부 작업 디렉터리를 지정합니다.

이후 명령어는 /app을 기준으로 실행됩니다.

 

COPY build/libs/app.jar app.jar
 
따라서 이 명령은 결과적으로 /app/app.jar 위치에 파일을 복사하게 됩니다.

 
COPY
 
COPY build/libs/app.jar app.jar
 
COPY는 호스트의 파일을 이미지 내부로 복사합니다.

여기서는 로컬의 build/libs/app.jar 파일을 이미지 내부의 /app/app.jar로 복사합니다.

즉 이미지 안에 내 애플리케이션 jar 파일을 넣는 과정입니다.

 
EXPOSE
 
EXPOSE 8080
 
EXPOSE는 이 컨테이너가 8080 포트를 사용할 예정이라는 정보를 이미지에 기록합니다.

하지만 중요한 점이 있습니다.

EXPOSE만 쓴다고 해서 호스트에서 컨테이너 포트로 접근할 수 있는 것은 아닙니다.

실제로 외부 접근을 열려면 docker run에서 -p 또는 --publish 옵션을 사용해야 합니다.

 
docker run -p 8080:8080 spring-app
 
Docker 공식 문서도 -p 또는 --publish 플래그를 사용해야 컨테이너 포트를 호스트 바깥에서 사용할 수 있게 된다고 설명합니다. -P는 Dockerfile의 EXPOSE나 --expose로 표시된 포트를 호스트의 임의 포트에 게시합니다.

 
ENTRYPOINT
 
ENTRYPOINT ["java", "-jar", "app.jar"]
 
ENTRYPOINT는 컨테이너가 시작될 때 실행할 기본 명령어입니다.

이 경우 컨테이너가 실행되면 다음 명령어가 실행됩니다.

 
java -jar app.jar
 
즉 Spring Boot 애플리케이션이 시작됩니다.

컨테이너는 이 메인 프로세스가 살아있는 동안 실행 중입니다. 때문에 이 프로세스가 종료되면 컨테이너도 종료됩니다.

DENTRYPOINT는 이미지의 메인 명령을 설정하는 데 적합하고, CMD는 그 명령의 기본 인자처럼 사용할 수 있어요.

 

정리
다시 간단하게 정리를 해볼게요.

 

이미지는 컨테이너를 만들기 위한 읽기 전용 템플릿입니다.

 
이미지 안에는 애플리케이션 실행에 필요한 파일, 라이브러리, 런타임, 설정, 실행 명령어 등이 들어갈 수 있습니다.

컨테이너는 이미지를 기반으로 실제 실행된 인스턴스이고요.

 
컨테이너는 격리된 파일 시스템, 네트워크, 프로세스 공간을 가집니다. 하지만 VM처럼 OS 전체를 새로 부팅하는 것은 아니고, 호스트 OS 커널을 공유하는 격리된 프로세스에 가깝습니다.

 

이미지는 읽기 전용이고, 컨테이너는 이미지 위에 쓰기 가능한 레이어를 추가해서 동작합니다. 컨테이너에서 만든 파일이나 수정 내용은 원본 이미지에 반영되지 않으며, 컨테이너를 삭제하면 그 쓰기 레이어의 데이터도 사라질 수 있습니다.

따라서 MySQL 같은 데이터베이스를 컨테이너로 실행할 때는 데이터를 컨테이너 내부에만 두면 안 됩니다. 데이터를 유지해야 한다면 volume을 사용해야 합니다.

 

핵심 흐름은 다음과 같아요.

Dockerfile
   ↓ docker build
Image
   ↓ docker run
Container
 

Docker 이미지와 컨테이너는 Docker의 가장 기본적인 개념이고, Docker를 깊게 이해하기 위해선 꼭 알아야 할 개념이기에 따로 정리를 해보았습니다. 추가적인 명령어에 대해선 다음 글에 정리해보고자 합니다. 긴 글 읽어주셔서 감사합니다.
